# 设计思路

## 一、设计理念

Next-Panel 的设计理念可概括为"**一个窗口管好所有服务器，一句话解决运维问题**"。我们将这一理念拆解为三个核心设计原则：

**原则一：零部署，即开即用。** 传统运维平台（如 JumpServer、Rancher）需要部署服务端组件、配置数据库、维护 Web 服务，部署成本高、学习曲线陡。Next-Panel 采用桌面应用形态，用户下载安装后即可使用——无需服务端部署、无需数据库配置、无需网络基础设施改造。所有数据本地存储，SSH 连接直连目标服务器，零中间环节。

**原则二：AI 原生，而非 AI 外挂。** 市面上的"AI 运维"产品大多是在现有工具上外挂一个 ChatGPT 对话框，AI 与运维操作是割裂的。Next-Panel 从架构层面将 AI Agent 引擎作为一等公民集成——AI 不是对话窗口里的聊天机器人，而是能直接操作远程服务器、采集系统数据、执行诊断命令的自治代理。

**原则三：人类在环，安全可控。** AI 的自动化能力必须始终在人类监督之下。本项目的所有 AI 远程操作均通过权限审批机制确认后才能执行，AI 提议、人类审批、系统执行——在自动化效率和操作安全之间取得最佳平衡。

## 二、架构设计思路

### 2.1 分层隔离架构

Next-Panel 采用四层隔离架构，每层职责明确、边界清晰：

**第一层：渲染进程（Presentation）**。基于 React 18 的组件化 UI 层，负责页面渲染、用户交互、动画效果。此层运行在 Chromium 沙箱中，通过 Preload Bridge 与主进程通信，无法直接访问 Node.js API 或文件系统。设计决策：选择 React 而非 Vue/Svelte，是因为 React 的 Hooks 模式更适合管理 Agent 的复杂异步状态（对话流、工具调用、权限等待）。

**第二层：Preload Bridge（Mediation）**。安全隔离层，定义了类型安全的 API 接口（window.electronAPI），包含 environment、terminal、chat、explorer、locale、theme、aiSettings 七大命名空间，共 40+ 个方法。设计决策：所有方法均为 Promise 化的异步调用，统一了同步/异步操作的编程模型。

**第三层：主进程（Business Logic）**。Electron 主进程承载所有业务逻辑——SSH 连接池管理、终端会话调度、SQLite 数据持久化、AI 配置管理。设计决策：选择 better-sqlite3 的同步 API 而非异步数据库，是因为主进程是单线程的，同步 API 避免了回调地狱且性能更优。

**第四层：AI Agent 引擎（Intelligence）**。独立的自治代理系统，运行在渲染进程内但逻辑上独立于 UI。Agent 引擎与 UI 通过 React Hook（useAgent）桥接——UI 触发对话，Agent 异步执行循环，通过状态更新驱动 UI 渲染。设计决策：Agent 运行在渲染进程而非主进程，是因为 AI 的流式输出需要直接驱动 DOM 更新，避免通过 IPC 传输大量流式数据的性能损耗。

### 2.2 数据流设计

系统的数据流遵循"单向数据流"原则：

```
用户操作 → React State 更新 → UI 重渲染
                ↓
        Preload Bridge IPC
                ↓
        主进程 Handler
                ↓
        SSH2 / SQLite / AI SDK
                ↓
        异步返回数据
                ↓
        React State 更新 → UI 重渲染
```

以 AI 对话为例的完整数据流：
1. 用户在 ChatInput 输入消息 → send() 调用 useAgent Hook
2. useAgent 构建 UserItem 加入 transcript，触发 UI 渲染用户气泡
3. useAgent 异步调用 loadAIConfig() 获取 API 配置
4. useAgent 调用 buildSystemPrompt() 动态构建系统提示词（注入 SystemSnapshot）
5. useAgent 调用 runAgent() 启动 Agent Loop
6. Agent Loop 调用 OpenAI SDK 的流式 Chat Completion API
7. SDK 通过 SSE 接收流式事件（content/thinking/tool_use/tool_result/done）
8. 每个事件触发 useAgent 的状态更新 → UI 实时渲染
9. 若遇到 tool_use 事件 → 暂停循环 → 显示 PermissionDialog → 等待用户确认
10. 用户确认后 → 通过 IPC 调用主进程的 terminal.exec 执行命令 → 返回结果
11. 结果注入 Agent Loop → 继续推理 → 生成最终回答

### 2.3 状态管理策略

Next-Panel 没有使用 Redux/Zustand 等全局状态管理库，而是采用"状态就近原则"——每个页面的状态由页面组件自己管理，跨页面共享的状态提升到 App 壳层。

**全局状态（App 层）**：
- `route: RouteId`：当前路由（dashboard/files/terminal/services/processes/firewall/cron/settings）
- `connection: { hostId, connectionId, status, error }`：当前活跃的 SSH 连接
- `drawerOpen: boolean`：AI 助手面板的展开/折叠状态
- `snapshot: SystemSnapshot | null`：远程主机系统快照（Dashboard 和 AI Drawer 共享）

**页面级状态**：
- Settings 页面：`hosts[]`、`form data`、`editing env`
- Files 页面：`currentPath`、`fileList[]`、`selectedFile`、`previewContent`
- Firewall 页面：`ufwStatus`、`nftRules[]`、`listeningPorts`

**SWR 缓存**：环境列表（hosts）使用 localStorage + 内存双层缓存，确保页面切换时数据即时展示。

## 三、AI Agent 设计思路

### 3.1 ReAct 模式选型

AI Agent 采用 ReAct（Reasoning + Acting）模式而非 Plan-and-Execute 或 Reflection 模式，基于以下考量：

运维场景天然适合"观察→分析→行动→观察"的循环模式。例如用户提问"服务器为什么卡"，AI 需要：先采集系统指标（SystemInfo 工具）→ 发现内存占用 95%（推理）→ 查看最占内存的进程（Bash 工具：`ps aux --sort=-%mem | head`）→ 发现是 Java 进程（推理）→ 检查 GC 日志（Bash 工具）→ 给出诊断结论和建议。

ReAct 的每一步都产生可观察的中间结果，这与运维场景要求的"可审计性"完美契合。

### 3.2 动态系统快照注入

传统 AI 对话的 System Prompt 是静态的（"你是一个运维助手"），AI 需要通过多轮问答才能了解用户的服务器环境。Next-Panel 创新性地在每次对话发起时自动注入动态上下文：

```
你是 Next-Panel 的运维 AI 助手。

当前服务器信息：
- 主机名：prod-web-01
- 操作系统：Ubuntu 22.04.3 LTS
- 内核：5.15.0-91-generic
- CPU：Intel Xeon E5-2680 v4 (8 核)
- 内存：27.3GB / 32GB (85.3%)
- 磁盘：412GB / 500GB (82.4%)
- 负载：4.23, 3.87, 3.52
- 运行时间：45 天

请基于以上实时数据回答用户问题。
```

这使得 AI 的首次回答即可基于准确的实时数据，无需用户手动描述服务器状况。

### 3.3 工具设计哲学

AI Agent 的工具设计遵循三个原则：

1. **最小权限原则**：每个工具只暴露完成其功能所需的最小参数集。Bash 工具只接受 `command: string` 一个参数，ReadFile 工具只接受 `path: string`，不多不少。

2. **幂等性优先**：SystemInfo 和 ListDir 是纯读取工具，天然幂等。Bash 工具的破坏性操作通过 Permission Dialog 拦截。

3. **可扩展性**：工具通过 JSON Schema 定义，新工具只需实现 `execute()` 方法和 Schema 声明即可注册到 Agent Loop，无需修改框架代码。

## 四、UI/UX 设计思路

### 4.1 1Panel 风格导航

Next-Panel 的导航设计参考了 1Panel（一款优秀的开源 Linux 服务器管理面板）的横向标签页模式。将模块导航从传统的左侧边栏提升为顶部 Pill 标签页，理由如下：

- 模块数量有限（8 个），标签页不会过度拥挤
- 横向导航释放了主内容区域的垂直空间
- 减少了鼠标移动距离——从侧边栏到内容区的斜向移动变为垂直移动
- 顶部导航栏同时承载品牌标识、模块导航、主机选择器、AI 开关四个功能区，信息密度高且层次清晰

### 4.2 AI Drawer 侧边抽屉

AI 助手面板采用右侧可抽拉式 Drawer 设计，而非全屏页面或底部面板。理由：

- 用户可以在查看 Dashboard/Files 的同时与 AI 对话，双屏并行
- Drawer 宽度固定（400px），不遮挡主内容区的核心信息
- 展开和收起的过渡动画（width transition）提供了空间感知

### 4.3 Design Tokens 设计系统

CSS 设计系统采用 Design Tokens 方法论，定义了完整的视觉语言：

- **颜色**：12 个语义化 CSS 变量（--bg、--surface、--accent、--danger 等），Dark/Light 主题仅需替换变量值
- **字体**：系统字体栈优先（SF Pro → Segoe UI → Helvetica），代码使用等宽字体栈
- **间距**：4pt 网格系统（--s-1=4px 到 --s-16=64px），确保视觉节奏一致
- **圆角**：6 级梯度（--r-xs=4px 到 --r-pill=9999px），组件层级越高圆角越大
- **阴影**：4 级层次（hairline → card → elev → pop），表达组件的 Z 轴层级

## 五、安全性设计思路

安全设计遵循"纵深防御"（Defense in Depth）原则，构建了五层安全防线：

1. **进程隔离**：contextIsolation=true、nodeIntegration=false，渲染进程无法访问 Node.js
2. **API 白名单**：Preload Bridge 仅暴露声明的 API，无法通过枚举发现未公开方法
3. **凭据加密**：SSH 密码与私钥采用 AES-256 对称加密存储，密钥派生自设备特征
4. **AI 权限审批**：所有 AI 远程操作需用户显式确认，前端 Dialog + 后端校验双重保障
5. **输入校验**：所有 IPC 边界的输入进行类型校验和合法性验证，拒绝异常数据
